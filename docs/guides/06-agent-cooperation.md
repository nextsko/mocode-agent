# 子代理协作实战笔记

> 整理时间：2026-06-12  
> 背景：本次孤儿代码扫描中，子代理报告有 **3 处事实错误**（callbacks.go / network.go / catwalk.go），差点导致严重删除事故

---

## 🎯 报告可信度分级

子代理报告的可信度从高到低：

| 可信度 | 类型 | 应对 |
|---|---|---|
| **高** | 文件存在/不存在（ls 结果）| 直接采纳 |
| **中** | 引用关系（grep 结果）| 二次核验 |
| **低** | "是否孤儿" 判断 | **必须**自己 grep 验证 |
| **极低** | "可以安全删除"建议 | **永远**自己 vet 一次 |

---

## ❌ 子代理的 3 类常见错误

### 1. 误报孤儿 — 把被 struct 字段引用的类型当孤儿

**案例**：`internal/config/network.go` 实际被 `config.go:269` 引用：
```go
Network *NetworkOptions `json:"network,omitempty" ...`
```

子代理报告"`NetworkOptions` 全 0 引用" — 实际是**通过字段类型**间接引用。

### 2. 漏报孤儿 — 漏掉 `//go:embed` 关联的目录

子代理可能不识别 `//go:embed` 指令，以为嵌入的文件/目录是孤儿。

**应对**：搜索 `//go:embed` 关键字逐个核验。

### 3. 错报关系 — 混淆了"包被 import"和"符号被引用"

子代理可能只看"包被 import"而忽略"符号被引用"。

## 🚨 本次会话的 3 个事实错误（教训）

| 子代理报告 | 实际情况 | 错误类型 |
|---|---|---|
| `callbacks.go` 0 引用，是孤儿 | 被 `agent.go:124, 147, 185, 230, 701` 引用 | 误报孤儿 |
| `network.go` 0 引用，是孤儿 | 被 `config.go:269 Network *NetworkOptions` 字段引用 | 误报孤儿 |
| `catwalk.go` 引用关系含糊 | `catwalkSync` 仅在 catwalk.go + 2 个 _test.go 中被引用 | 报告含糊 |

**关键教训**：永远不要盲信子代理报告，必须二次核验。

---

## 🛡️ 重点关注 3 类"假孤儿"

1. **通过 struct field type 引用**（如 `Network *NetworkOptions`）
2. **通过接口实现**（如 `var _ Interface = (*T)(nil)`）
3. **通过 init() 隐式注册**（如 cobra `rootCmd.AddCommand(...)`）

---

## ✅ 给子代理的精准任务卡（黄金模板）

```markdown
[Agent Task Card]
- Agent: <角色名>
- Mission: <单一任务目标>
- Inputs: <上游输出、参考文件>
- Constraints: 输出摘要而非全文
- Output Contract: <必须输出的字段和格式>
- DoD: <完成定义>
```

**实战教训**：本次用"列出 import 关系"的 prompt 没指定 Output Contract，导致子代理输出混乱、容易出错。**下次必须**用"返回去重排序的包路径列表"作为 Output Contract。

---

## 🔄 子代理报告的"乐观删除"陷阱

子代理倾向**先删再说**（"反正看起来是孤儿"），但删错一个非孤儿会导致编译失败、级联错误。

**应对**：任何"删除"动作前必须自己 grep 二次核验，并**先 git status** 确认基线。

---

## 🏗️ 子代理的 `//go:build ignore` 误读

子代理可能报告"`internal/db` 引用导致编译错误"，但**没注意到**被引用的文件都带 `//go:build ignore`。**gopls 也会持续报警告**（缓存延迟），造成假象。

**应对**：用 `view` 工具读前 5 行确认 build tag，**不要**信 gopls 的诊断。

---

## 📏 报告长度控制

| 报告长度 | 价值 | 应对 |
|---|---|---|
| < 50 行 | 信息密度高，但容易漏 | 自己补 grep |
| 50-200 行 | 平衡 | 选用 |
| 200-500 行 | 可能含噪声，需自己筛选 | 谨慎 |
| > 500 行 | 几乎一定有错 | 必有 1-2 处错误 |

**实战**：本次子代理报告约 100 行，但**关键 3 处错误**集中在最后 1/3（孤儿文件列表）。**教训**：报告里越后面的内容越要核验。

---

## 🧠 子代理的"凭印象断言"陷阱

子代理可能**凭印象**写"`agent.Callbacks` 全 0 包外引用"，没有真去 grep。实际 `agent.go:124` 有 `callbacks *AgentCallbacks` 字段。

**应对**：让子代理**贴出 grep 命令的输出**作为证据，而不是只给结论。

---

## 📋 子代理协作的 7 条铁律

1. **永远二次核验** — 任何"删除/修改"动作前自己 grep
2. **优先用结构化任务卡** — 黄金模板 + Output Contract
3. **子代理只做发现，不做决策** — 删除由人做
4. **报告长必有错** — > 500 行几乎一定有 1-2 处事实错误
5. **不信任"凭印象"** — 让子代理贴证据
6. **`//go:build ignore` 易误读** — 自己 view 文件确认
7. **删错成本远高于子代理节省的时间** — 保守为上

---

## 🗑️ 删孤儿代码的标准 5 步流程

```
1. 收集候选（grep 子代理）
2. 二次核验（自己 grep 关键候选）
3. 试探删除（先 git stash 备份，或在新分支）
4. 验证（go build / go vet / go test）
5. 正式 commit
```

**实战**：本次我们没做第 3 步（试探删除），但**通过二次核验**避免了删错 `callbacks.go` / `network.go`。
